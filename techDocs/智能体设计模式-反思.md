# 智能体设计模式 —— 反思（Reflection）

## 模式定义

反思（Reflection）是一种智能体设计模式，核心思想是：**给予agent完成任务后的反馈**，帮助其优化后续行为。

## 实践经验：Code Review 双Agent模式

### 协作流程
1. **Agent A（代码生成）**：负责编写代码
2. **Agent B（代码评审）**：从代码review角度给出反馈

### Agent B 评审维度
- **可读性**：命名规范、注释清晰、结构易懂
- **设计原则**：SOLID原则、DRY原则、单一职责
- **代码质量**：是否易于维护和扩展
- **潜在问题**：Bug风险、安全隐患、性能问题

### 价值体现
- 代码生成与评审解耦，各司其职
- 通过外部视角发现自身盲点
- 迭代优化，提升代码整体质量

### 扩展应用
此模式可推广到：写作→校对、策划→评审、方案→审查 等场景。

## 评估反思的影响

### 评估技巧：多元二元评判 > 单一打分

#### 核心观点
给出对结果的**多个二元评判标准**，往往比单一打分更有效。

#### 为什么？
- **更客观**：二元判断（是/否）比主观打分更明确
- **更具体**：每个维度独立评价，避免综合分数的模糊性
- **更易改进**：明确知道哪个维度不及格，针对性优化
- **减少偏见**：避免"还行"这种无法行动的模糊反馈

#### 示例对比
| 单一打分 | 多元二元评判 |
|----------|--------------|
| 整体评分：7/10 | 代码可读性好还是不好？✅ / ❌ |
|  | 是否满足设计原则？✅ / ❌ |
|  | 输出是否准确？✅ / ❌ |
|  | 性能达标：✅ / ❌ |
|  | 安全性：✅ / ❌ |

#### 实践建议
- 每个二元评判附上**简短理由**
- 维度数量控制在3-7个，避免过于冗长
- 定期统计各维度的通过率，识别共性问题

## 反馈来源

### 1. 自我反思
Agent 对自己的输出进行回顾和评价，识别不足之处。

### 2. 外部评价
- **其他 Agent 的评价**：由另一个agent对当前agent的输出提供反馈
- **执行结果反馈**：将任务执行的结果数据返回给agent，让其据此调整

## 应用场景

- 代码生成后的自我修正
- 回答生成后的质量检查
- 多Agent协作中的互相评审

## 核心价值

通过反馈闭环，让Agent能够：
- 识别错误
- 优化输出
- 持续改进

## 0样本提示模式（Zero-Shot Prompting）

### 定义
当你直接给Agent提出要求并让其实现时，这种方式称为 **0样本提示（Zero-Shot Prompting）**。

### "样本"的含义
样本（Shot）是指给予Agent的 **输入输出示例**：
- 输入示例：告诉Agent"问题是这样"
- 输出示例：告诉Agent"正确答案是这样"

### 有样本 vs 无样本
| 模式 | 说明 |
|------|------|
| **0样本（Zero-Shot）** | 无示例，直接下达指令 |
| **少样本（Few-Shot）** | 提供少量示例，让Agent学习模式 |
| **多样本（Multi-Shot）** | 提供多个示例，进一步强化理解 |

### 适用场景
- **0样本**：任务清晰明确，Agent能力足够
- **少/多样本**：任务复杂或输出格式有特定要求时

## 如何写好提示词：向优秀案例学习

### 核心方法
**阅读大佬写出的提示词**，从中学习结构、技巧和思路。

### 关键要素

#### 1. 给出更好的评估标准
- 明确"什么是好的输出"
- 定义清晰的评判维度
- 设定可量化的指标

#### 2. 更细节的要求
- **角色设定**：Agent应该扮演什么身份
- **任务背景**：为什么要做这件事
- **输出格式**：结构化输出，便于解析
- **边界约束**：明确禁止的行为

#### 3. 优秀提示词的特征
- 简洁清晰，避免歧义
- 任务拆解，步骤分明
- 提供必要上下文
- 包含示例（Few-Shot）

### 学习路径
1. 收集优质提示词案例
2. 分析其结构和技巧
3. 提炼可复用的模板
4. 根据任务类型适配调整
