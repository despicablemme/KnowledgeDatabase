## 1. 概述

### 1.1 业务背景

本团队正在自研大模型，模型效果**直接取决于训练语料的规模、质量与合规边界**。上游业务线掌握大量高价值原生语料（客服对话、产品文档、运营知识、用户行为日志等），是模型能力建设的重要来源。

然而，若直接摄入未治理的业务数据，会带来三类典型失败：

1. **合规失败**:含 PII、未授权采集、超出用户协议使用范围 → 监管处罚与品牌风险。
2. **质量失败**:噪声、口语病句、重复语料、模板化对话 → 模型学歪，越训越笨。
3. **污染失败**:测试集内容混入训练集、跨业务线内容串台 → 评测失真与能力偏差。

本规范即针对这三类失败设定前置治理标准。

### 1.2 目标

建立一套**面向大模型训练的上游数据接入契约**,使交付数据:

1. **合规** — 满足监管（个保法、数据安全法、生成式 AI 管理办法等）与内部合规策略。
2. **高质量** — 内容去重、去噪、合规脱敏、字段齐整、schema 稳定。
3. **可追溯** — 从生产环境到训练语料,每一条样本的来源、采集方式、处理动作均留痕。
4. **可验收** — 交付即通过验收,无需反复打回重做。

### 1.3 非目标（明确不做的事）

- **不**对业务线定义业务指标口径（KPI/DAU/留存等不是本规范范围）。
- **不**约束业务线内部用于自家业务分析的数据。
- **不**规定模型训练侧的数据配比、采样、过滤策略（训练侧自行决定）。
- **不**替代业务线自有合规流程,仅作为**对大模型交付的额外增量要求**。

## 3. 数据分级（敏感度）

业务数据进入大模型语料前必须先按敏感度分级。分级决定**是否可入训练 / 入哪种训练 / 走哪种通道**。

| 级别 | 名称 | 典型内容 | 可用范围 | 通道强度 |
| --- | --- | --- | --- | --- |
| **L0** | 公开 | 公开网页、新闻、书籍、官方文档、已脱敏操作手册 | 预训练 / SFT 全量 | 标准通道 |
| **L1** | 内部 | 产品内训手册、脱敏后工单、运营 SOP、合法采集的 FAQ | SFT / 知识库 RAG | 标准通道 |
| **L2** | 敏感 | 真实业务对话（已脱敏 PII）、用户咨询（已脱敏）、员工内部技术讨论 | SFT(受控) / 评测 | 加密 + 审计通道 |
| **L3** | 高敏感 | 财务、合同、未公开战略、未授权的内部研报 | **不可入训练**,仅用于评测分析 | 隔离区,不可入训练 |

### 3.1 分级判定流程

1. 业务线初判 → 填写 `data_level` 字段 → 自评依据留存。
2. 合规复核 → 对 L2/L3 出具结论。
3. 模型侧终评 → 对分级存在异议的数据由数据治理工作组仲裁。

> ⚠️ **分级误判责任在交付方**:分级过低导致合规事故的,由上游业务线承担主要责任。

### 3.2 不可入训练语料的内容（绝对红线）

下游数据工程师**必须**在交付前剔除以下内容(算法 + 人工双轨):

- 个人身份信息(PII):身份证号、手机号、银行卡号、邮箱、家庭住址、社保号、人脸/指纹生物特征、IMEI/MAC。
- 未成年人个人信息(不足 14 周岁)与未成年人对话内容。
- 未公开的财务、医疗、心理健康记录。
- 受版权保护的第三方原创内容(未经授权)。
- 受保密协议约束的内部机密(NDA 范围)。
- 已被监管或法律明确禁止用于训练的内容。

如不确定,默认**剔除**。可联系数据治理工作组白名单确认。

## 4. 合规与隐私要求

### 4.1 合规四要素

每条交付数据必须可回答下列四个问题(供合规审计):

1. **来源**:数据从哪个系统的哪条链路采集?
2. **授权**:用户是否在用户协议 / 隐私政策中同意该使用范围?
3. **手段**:采集、存储、加工、传输的方式是否合规?(加密、最小化、去标识)
4. **保留**:训练侧保留周期多长?是否会自动销毁?

### 4.2 授权最小化

训练语料遵循**目的限制 + 最小必要**:

- 仅为训练目的使用,**不**回流生产,**不**做原业务范围以外的分析。
- 默认 7 年保留周期(预训练权重类),到期自动触发销毁(见 §9.4)。

### 4.3 必要脱敏动作

L0/L1 数据可轻度处理(URL/邮箱脱敏);L2 必须做**强脱敏**:

| 处理项 | 规则 |
| --- | --- |
| 手机号 | 11 位中国大陆手机号 → `138****1234`;海外 E.164 按国家掩码 |
| 身份证号 | 18 位 → 前 6 后 4 中间全掩 |
| 银行卡号 | 仅保留后 4 位,**整段不进入语料** |
| 邮箱 | 用户名替换为 `user_<short_hash>` |
| 姓名 | 中文姓名 → `某先生 / 某女士` 或 `用户A`(按指令) |
| 物理地址 | 街道级以下地址 → 仅保留到地市 |
| 日期 | 时间戳 → 仅保留到日期(`YYYY-MM-DD`)用于时效性 |
| IP | 内外网 IP 全掩 |

> ⚠️ **审核要点**:脱敏后必须人工抽检 1% 验证无残留;验证人员名单与抽检记录留档。

### 4.4 合规交付前置物料

每条业务线**首次交付**与**重大变更**(协议变更、业务重大调整)时必须提交:

- 📄 《数据来源说明》:涉及系统、采集链路、字段清单
- 📄 《用户协议 / 隐私政策版本与生效日期》
- 📄 《合规评估意见》— 由业务线合规或本团队合规出具
- 📄 《数据保护影响评估》(DPIA) — 适用情形下(高敏感、跨境)

未提交上述物料的交付批次,默认**不接收**。

## 5. 数据质量要求

### 5.1 总体要求

进入训练语料的数据应在以下五个维度上达到门槛:

| 维度 | 含义 | 硬性指标 |
| --- | --- | --- |
| **完整性** | 必填字段缺失率 | < 0.5% |
| **去重率** | 近似重复样本占比 | < 0.5%(全局 MinHash) |
| **噪声率** | 无意义/损坏样本 | < 0.3% |
| **语言纯度** | 目标语言占比 | 单一语种 ≥ 95%(混合任务除外) |
| **有效性** | 用途/标签不符 | < 0.1% |

### 5.2 内容质量细则

- **语义完整**:不重复、不截断,首尾 6 字符不应为半句话。
- **语言规范**:
  - 中文任务:不混用繁体 / 简体(除非业务场景需要)
  - 英文任务:不混用英美拼写变体
  - 多语任务:必须标注主语言与脚本
- **去模板化**:`尊敬的客户您好,请问有什么可以帮您?` 类开场白 + `请问还有其他问题吗?` 类结束语,在 SFT 语料中**占比不得超过 5%**。
- **长度区间**:
  - 对话语料:5 ≤ 轮数 ≤ 50,单轮 8 ≤ 字数 ≤ 2000
  - 文档语料:200 ≤ 字数 ≤ 50000,保留文档结构
  - 工具调用语料:function name / 字段必须符合调用规范
- **标签/分类字段**:如果源数据已有标签,需校验标签分布的**类别均衡性**;极度不均衡(< 1%)的小类应在交付清单中说明。

### 5.3 字段命名与类型规范

为避免字段语义不一致,**统一字段字典**(见 §10.2)。通用必填字段在 §6.1 详述。

> 字段命名风格:snake_case,使用小写字母、数字、下划线;避免拼音首字母或缩写,必要时附中文释义。

### 5.4 元数据要求

每条样本必须携带以下元数据(用于追溯、回溯、复现训练):

| 元数据字段 | 是否必填 | 说明 |
| --- | --- | --- |
| `source_system` | ✅ | 上游系统名(如 `crm`, `ticket`, `kb` ) |
| `collection_method` | ✅ | 采集方式(`user_log_optin` / `synthetic` / `human_annotated`) |
| `data_level` | ✅ | L0 / L1 / L2(L3 不可入训练) |
| `pipeline_version` | ✅ | 业务线侧处理流水线的版本号(`vYYYYMMDD.x`) |
| `producer_id` | ✅ | 数据生产团队标识(`team_xxx`) |
| `produced_at` | ✅ | 生产时间(ISO8601,UTC) |
| `data_ttl_days` | ✅ | 该条数据声明的可保留天数 |
| `censor_state` | ✅ | `raw` / `regex_clean` / `ml_clean` / `human_reviewed` |
| `license_scope` | ✅ | 训练可用授权范围,如 `train_sft_only` |
| `train_split_hint` | 否 | `train` / `dev` / `test` / `unspecified` |

## 6. 交付格式与 Schema

### 6.1 通用 Schema(JSON Lines / JSONL)

**首选格式**:`.jsonl`,每行一条 JSON;UTF-8,LF 换行。亦可使用 Parquet,但提交前**同步提交 JSONL 副本**用于校验。

每行 JSON 至少包含 `text` 与 `meta`,推荐结构:

```json
{
  "id": "c2c-2026-07-14-00001",
  "text": "用户: ...  助手: ...",
  "meta": {
    "source_system": "customer_service",
    "collection_method": "user_log_optin",
    "data_level": "L1",
    "pipeline_version": "v20260705.3",
    "producer_id": "team_support_ai",
    "produced_at": "2026-07-12T08:33:17Z",
    "data_ttl_days": 2555,
    "censor_state": "human_reviewed",
    "license_scope": "train_sft_only"
  },
  "task_type": "open_qa",
  "tags": ["售后", "退款"],
  "messages": [
    {"role": "user", "content": "我的订单还没收到"},
    {"role": "assistant", "content": "请提供订单号,我在帮您查询"}
  ]
}
```

### 6.2 字段语义

| 字段 | 必填 | 含义 |
| --- | --- | --- |
| `id` | ✅ | 业务线内全局唯一 ID;**跨交付批次不可重复**;建议 `<业务线>-<日期>-<序号>` |
| `text` | ✅ | 全文原文(供预训练) |
| `messages` | ❓(多轮必填) | 结构化对话;`role ∈ {system, user, assistant, tool}`,`content` 为字符串或内容数组 |
| `task_type` | ❌ | 任务类型枚举,参考 §10.3 字典 |
| `meta` | ✅ | 元数据,见 §5.4 |
| `tags` | ❌ | 业务侧标签,用于数据子集筛选 |
| `quality_score` | ❌ | 0-1 质量评分;不强制但推荐 |

### 6.3 目录结构与命名

```
交付根目录/
├── manifest.json                # 交付清单
├── README.md                    # 业务侧自述
├── data/
│   ├── part-000.jsonl.gz
│   ├── part-001.jsonl.gz
│   └── ...
├── schema/
│   └── fields.dict.json
├── license/
│   └── LICENSE-授权说明.md
├── compliance/
│   ├── DPIA-2026-07-12.md
│   └── 合规评估意见.md
└── sample/
    └── sample-100.jsonl         # 100 条样本预览,仅用于核对格式
```

- **目录命名**:`<业务线>_<数据集>_<日期>`;如 `support-faq_v3_2026-07-12`。
- **批次大小**:单批压缩包 ≤ 50 GB,文件数 ≤ 1000。
- **压缩**:建议 `.gz`,禁止 `.zip`(避免编码歧义)。

### 6.4 增量 vs 全量

| 类型 | 含义 | 命名后缀 |
| --- | --- | --- |
| **全量**(snapshot) | 截止 `produced_at` 的全量数据 | `_full_YYYYMMDD` |
| **增量**(delta) | 相对上次交付的增量 | `_delta_YYYYMMDD_HHmm` |

> 增量数据**必须**包含 `delta_op ∈ {insert, update, delete}`,便于模型侧 upsert。

## 7. 交付流程与通道

### 7.1 交付节奏

| 节奏 | 适用业务线 | SLA |
| --- | --- | --- |
| **月度全量 + 周增量** | 通用业务(SFT/知识库) | 月全量 5 个工作日内增量合并 |
| **周全量** | 客服/对话类 | 每周末 |
| **事件触发** | 应急演练 / 法务整改 | 24 小时 |

### 7.2 交付通道(任选其一,但需事先签批)

1. **内部对象存储**(主用):业务线上传至 `s3://jarvis-data-landing/<业务线>/<数据集>/`,通过临时 URL 授信签名。
2. **Kafka 主题**:适用实时流式语料;topic 命名 `<business>-<dataset>`;分区数 ≤ 64,保留期 ≤ 72h。
3. **离线硬盘**:不推荐;需要数据治理工作组现场审批。

### 7.3 交付前置 Checklist

业务线数据负责人交付前必须逐条勾选:

- [ ] 数据分级字段已填写
- [ ] 合规前置物料已提交(首次 / 变更时)
- [ ] 字段字典已与 `fields.dict.json` 对齐
- [ ] 100 条样本预览已生成,无字段错误
- [ ] 通过 §5 全部硬性指标(自检)
- [ ] `manifest.json` 已填写,见 §10.1
- [ ] 业务线内部合规 / 法务已书面签字

### 7.4 SLA 与故障处理

| 指标 | 目标 | 失效处置 |
| --- | --- | --- |
| 交付及时率 | ≥ 99%(月) | 一次延迟:T+1 给原因;连续两月:升级 |
| 验收一次合格率 | ≥ 95%(月) | 低于目标:专题复盘 + 改进措施 |
| 通道可用率 | ≥ 99.5% | 故障 1h 内通报,4h 内恢复或迁移 |

## 8. 验收标准

### 8.1 验收方式

采用 **自动化质检 + 人工抽检**双轨:

1. **自动质检**(≥ 99% 通过)运行 §10.4 全部脚本。
2. **人工抽检**(1% 样本,≥ 200 条)由数据治理工作组负责。
3. **专项复检**:合规事件发生后触发 100% 全量回扫。

### 8.2 验收结论

| 结论 | 含义 | 下一步 |
| --- | --- | --- |
| ✅ **PASS** | 完全符合规范 | 进入训练流水线 |
| ⚠️ **PASS-WITH-NOTE** | 通过,带改进建议 | 入训 + 业务线下一批次整改 |
| ❌ **FAIL**(机制) | 字段、格式问题 | 业务线 3 个工作日内返修 |
| ❌ **FAIL**(合规) | 合规风险 | 数据治理工作组冻结该批,**启用事件响应**(§9.5) |

### 8.3 返修流程

- 业务线收到 `FAIL` 通知后,**不超过 3 个工作日**提交新批次。
- 返修率连续 3 次 ≥ 20%,触发业务侧数据治理复盘。

## 9. 安全与生命周期

### 9.1 存储加密

- 传输过程:SSL/TLS 1.2+ 强制。
- 落地加密:AES-256,密钥由 KMS 统一管理。
- L2 及以上数据落地**必须**加密 + 启用静态脱敏。

### 9.2 访问权限

- 最小权限原则(Just-In-Time, Just-Enough-Access)。
- 训练侧工程师仅可见已脱敏数据;原始数据需经合规审批。
- 模型训练日志、外部调试导出**不得**含原始语料。

### 9.3 审计日志

- 任何数据访问、查询、下载均产生不可篡改的审计日志。
- 字段:`who`、`when`、`what`、`reason`、`data_id`。
- 保留周期不少于业务侧 TTL + 6 个月。
- 异常访问模式(高频下载 / 跨业务线访问)自动触发告警。

### 9.4 数据销毁与退场

- 模型侧按 `data_ttl_days` 字段自动从训练数据池销毁。
- 销毁动作产生日志,**不可逆**。
- 业务侧主动退场:在 `manifest.json` 提交 `revoke` 操作,适用 delete-op 三日回放 + 最终清理。

### 9.5 合规事件响应

发生疑似/确认合规事件(PII 泄漏、未授权使用等):

1. 业务侧 1 小时内通报数据治理工作组。
2. 数据治理工作组 4 小时内启动 **STOP-TRAIN** 评估;评估通过则暂停该数据流。
3. 24 小时内合规 / 法务给出事件评级与处置预案。
4. 5 个工作日内给出根因报告与改进计划。

### 10.4 自动质检脚本清单

下列脚本由数据治理工作组维护,纳入交付验收流水线:

| 脚本 | 检查项 |
| --- | --- |
| `check_schema.py` | 字段完整性、类型、enum 范围 |

| `check_language.py` | 语言识别与目标语言纯度 |

| `check_meta.py` | 元数据必填字段与时间格式 |
| `check_balance.py` | 分类标签分布均衡性 |
| `check_format.py` | JSONL 解析、UTF-8、LF 换行 |
