# 记账技能 (Accounting)

## 核心原则

**单一数据源**：所有数据从 `transactions` 表出发，余额和汇总实时计算，不再有派生表导致的数据不一致问题。

---

## ⚠️ 重要规则

### 1. 禁止手动修复数据库
**任何情况下都不准手动 sqlite3 操作数据库**，所有数据操作必须通过脚本完成。

### 2. 撤回交易的方式
如果需要撤回一笔收入或支出，**不能删除记录，必须追加一笔负数**：

```bash
# 撤回某笔支出（如晚饭外卖57.4）
python3 scripts/accounting.py expense 57.4 撤回晚饭外卖 饮食

# 撤回某笔收入（如工资8000）
python3 scripts/accounting.py income 8000 撤回工资
```

这样做的原因：
- transactions 是唯一数据源，余额和汇总都从它实时计算
- 删除记录会打乱 id 自增序列，且汇总表无法同步删除对应的汇总数据
- 追加负数可以完整保留交易链痕迹，余额自动抵消，汇总表同步更新

### 3. 汇总表单入口原则
**三个汇总表（周/月/年）必须从同一个入口更新，不允许单独修改任何一个汇总表。**

代码实现上：
- `record_income()` 和 `record_expense()` 在同一个事务内调用三个 `upsert_*_summary()` 函数
- 这三个函数是并列的，不是串行的独立逻辑
- 任何新增/修改汇总更新的逻辑，必须同时处理周/月/年三个表

```
BEGIN TRANSACTION
  → INSERT transactions
  → upsert_weekly_summary()
  → upsert_monthly_summary()
  → upsert_yearly_summary()
COMMIT
```

## 数据库结构

数据库位置：`~/.openclaw/workspace/skills/accounting/scripts/accounting.db`

| 表名 | 主键 | 用途 |
|------|------|------|
| `transactions` | id | 收支明细（唯一数据源） |
| `weekly_summary` | (year, week) | 周收支汇总 |
| `monthly_summary` | (year, month) | 月收支汇总 |
| `yearly_summary` | (year) | 年收支汇总 |

### transactions 表
```sql
id INTEGER PRIMARY KEY,
type TEXT,        -- 'income' 或 'expense'
amount REAL,      -- 金额
category TEXT,    -- 分类（支出时由AI判断，收入时为"收入"）
description TEXT, -- 描述
date DATE,        -- 日期
created_at TIMESTAMP
```

### 汇总表结构
```sql
-- weekly_summary / monthly_summary / yearly_summary
year INTEGER,
week/month/year INTEGER,
total_income REAL,        -- 该周期总收入
total_expense REAL,       -- 该周期总支出
category_expenses TEXT,   -- JSON，格式：{"分类名": 金额, ...}
-- 周报还有 week_start, week_end
```

## AI 分类流程（关键）

**支出记录的分类不是靠关键词匹配，而是由我（AI）理解描述后判断。**

### 处理流程

1. **接收支出信息**：用户说"买衣服花了200"
2. **理解语义**：我判断这是"穿戴类消费" → 分类为"购物"
3. **调用脚本**：`expense 200 买衣服 购物`
4. **脚本写入**：transactions + 三个汇总表（同一事务）

### 预设分类

| 分类 | 适用场景举例 |
|------|------------|
| 饮食 | 外卖、餐厅、零食、买菜、饮料 |
| 娱乐 | 电影、游戏、音乐、演出、旅游、KTV |
| 学习 | 书籍、课程、培训、订阅、技术资料 |
| 交通 | 地铁、打车、公交、高铁、机票、油费 |
| 购物 | 衣服、鞋包、化妆品、电器、日用品、网购 |
| 医疗 | 医院、药品、体检、牙科、药店 |
| 住房 | 房租、水电、物业、房贷、装修 |
| 通讯 | 手机话费、网络宽带、流量 |
| 社交 | 请客、送礼、红包、份子、聚会 |
| 运动 | 健身、跑步、游泳、瑜伽 |
| 其他 | 以上都不适合的 |

### AI 判断原则

- 根据描述的**语义**判断，不是死板匹配关键词
- 遇到新描述：理解其本质，归入最合适的分类
- 分类是动态的，如果有新场景，可以新建分类

## 余额计算

**余额 = 所有收入之和 - 所有支出之和**，实时从 transactions 计算：

```sql
SELECT COALESCE(SUM(
    CASE WHEN type='income' THEN amount ELSE -amount END
), 0)
FROM transactions
[WHERE date <= '日期']
```

不存单独的余额表，每次按需计算。

## 支出分布比例

每个周期的支出分布都从汇总表的 `category_expenses` JSON 字段读取：

```python
categories = json.loads(category_expenses)  # {"饮食": 110.79, "交通": 13.40}
# 显示时：
for cat, amt in sorted(categories.items(), key=lambda x: -x[1]):
    pct = amt / total_expense * 100
    bar = '█' * int(pct / 5) + '░' * (20 - int(pct / 5))
    print(f"{cat}: ¥{amt:.2f} ({pct:.1f}%) {bar}")
```

## ⚠️ 血泪教训（2026-05-20）

### 教训1：用户说"帮我算一下"不是要记账

**场景：** 用户说"午饭开销是69除以4"，意思是让我帮他算数学题（69÷4=17.25），不是要我记账。

**错误：** 我误以为他要记账，直接调用了 `expense` 命令，导致多记了4笔晚饭支出。


**正确做法：**
- 用户明确说"记账"、"记一笔"、"记下来"时才记录
- 如果只是"帮我算一下"、"多少钱"等，纯计算即可，**不要**自动创建记录
- 除非用户明确要求，否则不要擅自记录

### 教训2：用户纠正时要先查实再行动

**场景：** 用户说"截止昨天我的余额是-361.81"，我直接相信并尝试用收入冲抵，导致余额越冲越乱。

**正确做法：**
1. **先用 `balance` 命令查证用户的说法**
2. 确认用户说的余额和系统一致后，再分析差额原因
3. 查证是美德，不要急于修复而不验证

### 教训3：撤回不能解决问题，要用收入冲抵

**场景：** 我误用 `expense 14 撤回晚饭` 来"撤回"记录，结果只是又记了一笔14元支出，余额更负了。

**正确做法：**
- 撤回的正确方式是：**追加一笔同等金额的收入**（如果是误记支出）或**追加一笔同金额的支出**（如果是误记收入）
- `expense 14 撤回xx` 只是记录了一笔新的14元支出，不是撤回
- 本质是：余额 = 收入 - 支出，所以要平衡余额需要用反方向的交易

### 教训4：修复后立即验证余额

**场景：** 我冲抵了4笔14元的误记后，余额是 -550.31，而不是预期的 -361.81 - 188.50 = -550.31。

**正确做法：**
- 修复完成后，**立即用 `balance` 命令核验**
- 用数学验证：截止昨天的余额 ± 今天的新记录 = 当前余额
- 只有验证通过，才能告诉用户数据库正确了

### 教训5：记录要精准，不要批量操作

**场景：** 我一次性调用了4次 `expense 14 撤回id21` 等命令，想批量撤回，结果自己也搞混了。

**正确做法：**
- 操作要**逐笔确认**，每完成一步验证余额后再进行下一步
- 批量操作容易出错，出错后更难追踪
- 如果必须批量，操作前先列出计划，让用户确认后再执行
